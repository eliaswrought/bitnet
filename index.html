the thesis

Bitcoin created the first decentralized market: miners provide hashpower to secure the chain and receive bitcoin in return. No coordination required. No business development. Pure incentive drove it from one node to a global network.

Lightning created the second: node operators provide liquidity and routing capacity, earning fees on every payment they forward. Again ‚Äî permissionless participation, organic growth through economic incentive.

Both markets share five critical properties:

  • Discrete ‚Äî one unit of work, one unit of payment
  • Simple ‚Äî the protocol defines everything, no negotiation
  • Instantly verifiable ‚Äî proof of work is binary, routing either succeeds or fails
  • Permissionless ‚Äî anyone can start participating today
  • Self-reinforcing ‚Äî more participants = better network = more participants

Bitnet applies this exact pattern to the most fundamental internet service: moving data from point A to point B.

The core claim: A decentralized market for bandwidth relay, settled in bitcoin over lightning, can grow organically on incentives alone ‚Äî just as mining and routing did. No token. No foundation. No fundraise. Just software that makes it profitable to relay traffic.

Every other internet service ‚Äî storage, compute, VPN, CDN ‚Äî depends on data transport. Bandwidth relay is the primitive. Build this market first, and the others can layer on top.

how it works

A lightweight daemon called bitnetd runs alongside any Lightning node. It does two things: accepts incoming proxy requests, and charges sats to fulfill them.

Request Lifecycle

Fig 1. Single-node request lifecycle ‚Äî payment is atomic via Lightning HTLC

The critical property: payment and delivery are atomic. The client pays via a Lightning HTLC (Hash Time-Locked Contract). The node only claims payment after delivering the response. If relay fails, the payment reverts. No trust required.

Multi-Hop Relay

For privacy, requests can be routed through multiple bitnet nodes using onion routing ‚Äî the same technique Lightning itself uses for payment routing. Each node only knows the previous hop and the next hop. Nobody sees the full path.

Fig 2. Multi-hop onion relay ‚Äî each node earns fees, none sees the full path

The payment splits across all relay nodes in the path, just as Lightning routing fees split across routing nodes. Exit nodes charge more (they take the legal exposure of touching the public internet). Entry and middle nodes charge less.

the protocol

L402: Lightning HTTP 402

Bitnet uses L402 (formerly LSAT) ‚Äî a protocol developed by Lightning Labs that combines macaroons (bearer authentication tokens) with Lightning invoices. When a client requests relay service without a valid token, the node responds with HTTP 402 Payment Required, including a macaroon and a Lightning invoice.

HTTP/1.1 402 Payment Required
WWW-Authenticate: L402 macaroon="AGIAJEem...", invoice="lnbc200n1p..."

# Client pays the invoice, obtains preimage
# Presents macaroon + preimage as valid L402 token
# Node verifies cryptographically ‚Äî no database lookup needed

Authorization: L402 AGIAJEem...:preimage=79852a07...

Key properties of L402 for bitnet:

  • Stateless verification ‚Äî any node can verify a token with just the root key. No central database of users or invoices.
  • Metered access ‚Äî macaroons can encode caveats: bytes remaining, expiry time, allowed services. A token might grant 10 MB of relay or 30 minutes of access.
  • Delegatable ‚Äî tokens can be attenuated (restricted further) and passed to others. A client could buy a large relay allocation and subdivide it.
  • No identity ‚Äî L402 tokens prove payment, not identity. No email, no account, no registration.

Nodes advertise their pricing. Clients choose nodes based on price, latency, and location. The market sets the rate.

Node Discovery

Bitnet nodes announce their presence and capabilities through the existing Lightning gossip protocol. Each node publishes:

  • Relay capacity ‚Äî available bandwidth (Mbps)
  • Pricing ‚Äî sats per MB, per request, or per time window
  • Location ‚Äî approximate geographic region (opt-in)
  • Services ‚Äî HTTP proxy, SOCKS5, TCP relay, exit node (opt-in)
  • Uptime history ‚Äî rolling average availability

Clients maintain a local view of available relay nodes and select paths based on their requirements (cheapest, fastest, most geographically diverse, etc.).

why lightning nodes are day-one infrastructure

The cold start problem kills most decentralized networks. Bitnet doesn't have one.

There are approximately 15,000 Lightning nodes running globally right now. Every single one of them already has:

  • A public IP address ‚Äî they're reachable from the internet
  • Spare bandwidth ‚Äî Lightning payments are tiny; most node bandwidth is idle
  • A Lightning wallet ‚Äî ready to generate invoices and receive payments
  • 24/7 uptime ‚Äî nodes must stay online to route payments and monitor channels
  • Existing connectivity ‚Äî channels already open to other nodes worldwide
Fig 3. A Lightning node already has everything bitnet needs ‚Äî just activate the idle bandwidth

The installation pitch: You already run a Lightning node. Install one binary. Earn sats from your spare bandwidth. Takes 5 minutes.

This is not "build a new network from scratch." This is "turn on a feature on a network that already exists." Day one, bitnet has access to thousands of globally distributed, always-on, payment-ready nodes. No other decentralized bandwidth project can claim this.

the organic growth flywheel

The growth mechanism mirrors mining and routing exactly: economic incentive drives participation, participation improves the network, network improvement attracts users, users generate revenue, revenue attracts more participants.

Fig 4. The growth flywheel ‚Äî same incentive loop as mining and routing

No convincing needed. Node operators don't join because of ideology or altruism. They join because running bitnetd costs them nothing (spare bandwidth on hardware they already run) and earns them sats. The network gets better as a side effect of everyone pursuing their own economic interest.

This is Adam Smith's invisible hand applied to internet infrastructure.

first use cases

1. Censorship Circumvention

People in Iran, China, and Russia currently pay for VPN subscriptions that get blocked within weeks. Bitnet is different: nodes appear faster than censors can block them, because anyone with a Lightning node can become a relay. There's no company to sanction, no central server to shut down, and the payment itself is uncensorable (Lightning). The first real users will be people who need uncensored internet and already have access to bitcoin.

2. Privacy Browsing Without Subscriptions

Pay 50 sats, browse for 10 minutes through an encrypted relay. No account, no email, no credit card linking your identity to your traffic. No monthly commitment. Use it once or use it daily ‚Äî the network doesn't know or care.

3. Machine-to-Machine API Proxying

Bots and agents that need to make API calls without revealing their origin IP, rotate IPs for scraping, or access geo-restricted data. Pay per request, programmatically, with no human in the loop. This is where Lightning's programmable money becomes essential ‚Äî machines paying machines for bandwidth.

4. Decentralized File Transfer

Route a file transfer through the relay network. End-to-end encrypted, paid per byte, no server in the middle. No Dropbox, no Google Drive, no WeTransfer. Just two wallets and a swarm of paid relays.

5. Developer IP Rotation

Developers who need rotating IPs for testing, monitoring, or data collection. Instead of buying proxy lists from sketchy providers, point your code at bitnet. Pay-per-request, clean IPs, no monthly contract.

The wedge: Censorship circumvention is the initial market because it has inelastic demand ‚Äî people in authoritarian countries don't stop needing uncensored internet because the price went up. And their current solutions (centralized VPNs) keep getting worse.

The Gap

The market is validated ‚Äî multiple well-funded teams have attempted to build decentralized bandwidth markets. But every one of them introduced a custom token, which:

  • Adds friction (users must acquire a new token before they can use the network)
  • Fragments liquidity (relay fees denominated in an illiquid altcoin)
  • Introduces speculation (token price disconnects from service value)
  • Centralizes governance (token holders ‚â† network participants)

Tor proved the demand exists but showed that donation-based models produce slow, underfunded networks. Impervious proved Lightning can work as a data transport layer but tried to build too much at once.

Bitnet's thesis: The right money already exists. You don't need a new token to pay for bandwidth. You need sats. Strip the altcoin, use Lightning, and leverage the 15,000 nodes that already form a global payment network.

technical architecture

Fig 5. System architecture ‚Äî bitnetd connects to the local Lightning node and serves relay clients

Components

bitnetd ‚Äî The relay daemon. Written in Go for ecosystem compatibility with lnd. Connects to the local Lightning node via gRPC for invoice generation and payment verification. Runs an L402-gated proxy server (SOCKS5 and HTTP). Participates in node discovery via Lightning gossip extensions. Lightweight ‚Äî minimal CPU and memory footprint on top of an existing Lightning node.

bitnet-cli ‚Äî The client tool. Routes local traffic through one or more bitnet nodes. Handles L402 negotiation, automatic invoice payment, and path selection. Can be used as a system-wide proxy or per-application.

Browser extension (future) ‚Äî One-click "connect to bitnet" for non-technical users. Automatically finds the cheapest/fastest relay, pays invoices from an embedded Lightning wallet, and routes browser traffic through the network.

SDK / Library (future) ‚Äî For developers integrating bitnet relay into their applications. Pay-per-request API proxying, bot traffic routing, programmatic bandwidth purchase.

legal considerations

The primary legal question is the same one Tor exit node operators have faced for 20 years: if you relay someone else's traffic, and that traffic is illegal, are you liable?

Existing Protections

  • DMCA ¬ß512 / Safe Harbor ‚Äî Intermediaries that transmit data without modification have legal protection. ISPs relay illegal traffic daily and aren't liable. Bitnet nodes function identically to an ISP or Tor relay.
  • Section 230 ‚Äî Protects intermediaries from liability for content passing through their systems.
  • Common Carrier Doctrine ‚Äî If you relay everything indiscriminately (no content selection), you're a conduit, not a publisher.

Risk Mitigation

  • Relay-only by default. Bitnet nodes do not act as exit nodes unless explicitly opted in. Default behavior is middle-relay only: pass encrypted traffic to the next node. Only operators who choose to accept the additional risk (and earn higher fees) become exit nodes.
  • Commercial conduit framing. Lightning payments make bitnet nodes commercial relay services, closer to ISPs than volunteer projects. ISPs have stronger legal protections than volunteer anonymity networks.
  • No content awareness. Nodes relay encrypted blobs. They have no ability to inspect, filter, or modify content ‚Äî strengthening the common carrier argument.
  • Open protocol, not a service. Bitnet publishes software, not a service. The protocol is open. Each node operator runs their own independent instance. This mirrors BitTorrent's legal position.

Jurisdictional Notes

The US offers the strongest protections for relay operators. Some jurisdictions (notably Germany) have been historically hostile to Tor exit operators. Bitnet documentation should include clear guidance on jurisdictional risk and recommend relay-only mode in higher-risk jurisdictions.

Establishing a legal defense fund as the network grows would mirror the EFF's support for Tor operators and provide additional protection for participants.

roadmap

Phase 1

Single-Node Proxy

One node running bitnetd alongside a mainnet Lightning node. L402-gated HTTP proxy. Clients pay per request or per time window. Proof of concept ‚Äî demonstrate the core payment-for-relay loop works end to end.

Phase 2

Multi-Node Relay

Node discovery via gossip. Multi-hop routing with onion encryption. Client path selection. Fee splitting across relay nodes. SOCKS5 support. This is where bitnet becomes a network rather than a single proxy.

Phase 3

Client Ecosystem

Browser extension for one-click relay. Mobile client. SDK/library for developer integration. Focus on UX ‚Äî make it trivial for non-technical users to route traffic through bitnet.

Phase 4

Network Dashboard

Public visualization of the relay network. Live node map, traffic stats, earnings leaderboard. Marketing flywheel ‚Äî operators share their node's position and earnings, attracting more participants.

Phase 5

Exit Nodes + Ecosystem

Opt-in exit node support with clear legal documentation. Legal defense fund. QoS tiers (pay more for low-latency or high-bandwidth relay). TCP/UDP relay beyond HTTP. Foundation for higher-layer services (storage, compute).

the pattern

Bitcoin proved that you can build a global network purely on incentives. No company needed to organize miners. No foundation needed to fund them. The protocol defined the reward, and rational actors did the rest.

Lightning proved the same pattern works for a second market ‚Äî routing payments. Node operators provide liquidity because it's profitable, and the network grows as a consequence.

Bitnet is the hypothesis that this pattern works for a third market: bandwidth. The most fundamental internet service, settled in the most neutral money, over the fastest payment network in existence.

No token. No foundation. No fundraise. Just software that makes it profitable to relay traffic for strangers.

The network is the incentive. The incentive is the network.
